# TKT-050: Define semantic chunk ID format

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-050` before committing. -->

## Description
Why: incremental upserts and deletes require stable chunk identifiers.

Scope:
- Define a deterministic chunk ID format.
- Encode ticket ID, field identity, and repeated-item position where needed.

Files to touch:
- Semantic chunking package and unit tests.

Non-goals:
- No hash algorithm or storage schema.

## Acceptance Criteria
- [x] Each semantic chunk gets a deterministic ID that remains stable across rebuilds when ticket content is unchanged. : Chunk IDs are deterministic and encode ticket ID, chunk type, and AC position where needed.
- [x] The ID format can distinguish ticket, field, and repeated item position without relying on database row IDs. : Chunk tests verify stable IDs across repeated chunk generation.

## Handoff
**Current state:**
Chunk IDs are stable across rebuilds and distinguish repeated AC items.

**Decisions made:**
Use readable string IDs like ticket:type and ticket:ac:index rather than row IDs or opaque hashes.

**Files touched:**
- internal/semantic/chunk.go
- internal/semantic/chunk_test.go

**Remaining work:**
- Use chunk IDs as the stable key for metadata and vector-store operations later.

**AC status:**
- Both acceptance criteria completed and covered by tests.
