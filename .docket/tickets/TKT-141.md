# TKT-141: Spike: multi-repo support in docket ui via shared port and project selector

## Description
Investigate and prototype support for running a single docket ui server that can serve multiple repos simultaneously, with a project selector in the UI.

## Context

Currently each `docket ui` launch starts a new SvelteKit dev server bound to a single DOCKET_DIR. When multiple repos have `docket ui` running, they each occupy a separate port with no cross-project visibility.

## Goal

Evaluate feasibility of a single shared port acting as a hub, with a project selector dropdown in the UI to switch between registered repos.

## Approaches to evaluate

### Option A: Hub server with in-process registry
- First `docket ui` call starts the server
- Subsequent calls from other repos POST to `/api/projects/register` instead of launching a new process
- Server maintains an in-memory (or file-backed) registry of project dirs
- UI shows a project picker, switching via `?project=<id>` query param

### Option B: Lockfile-based registry
- Each `docket ui` call writes its DOCKET_DIR to `~/.docket/ui-projects.json`
- A single long-lived server (started once, possibly via `docket ui --daemon`) reads that file on each request
- No inter-process communication needed

## Areas to investigate
- Which approach is simpler to implement given the SvelteKit dev server architecture
- How to handle server restarts (re-registration, lockfile cleanup)
- Impact on `/api/tickets/[id]` mutation handler (currently uses DOCKET_DIR from env)
- Whether to use dev server or switch to build+preview for the multi-tenant use case

## Deliverable
A recommendation (Option A, B, or hybrid) with a concrete implementation plan and estimated scope.
