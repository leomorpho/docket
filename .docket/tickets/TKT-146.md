# TKT-146: Optimize MCP tools for large payloads via temporary files

## Description

Address the "Permission Prompt" and shell argument length limits when LLMs pass massive descriptions or handoffs to Docket. Instead of forcing the LLM to pass thousands of characters in a single tool call, provide a mechanism for the agent to write the content to a temporary file first, which Docket then reads.

## Acceptance Criteria

- [x] Update MCP tools `create_ticket` and `update_ticket` to accept an optional `content_file` (string) field.
- [x] If `content_file` is provided, the MCP server must read the file and use its content as the ticket description/handoff.
- [x] Implement a safe reading mechanism that validates the path (e.g., must be within the repo or a known temp dir).
- [x] Update the Docket Skill (TKT-144) and Cursor rules (TKT-145) to instruct agents: "If your content is > 1000 characters, use a temporary file for the update."

## Technical Context

- This solves the security heuristics problem where agents prompt users for permission when commands look "too large."
- It also improves reliability for very large handoff summaries generated by `session compress`.

## Plan

1. Modify `internal/mcp/` logic to handle the `content_file` parameter.
2. Update tool definitions to include this new optional parameter.
3. Add a test case that creates a ticket from a temporary file.
