# TKT-174: [Epic] Govern workflow policy and privileged state transitions

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-174` before committing. -->

## Description
This epic turns the workflow discussions into enforceable product behavior while keeping the developer experience strong during rollout.

Execution order:
1. Make workflow semantics config-defined and remove hardcoded policy assumptions.
2. Land the simple warning-only versus enforced rollout for secure behavior.
3. Make CLI, MCP, status, doctor, and setup surfaces consistent with that rollout.
4. Layer approval, activation, and rollout guidance on top without changing the two-state DX model.

Scope:
- Separate editable workflow proposals from approved runtime policy.
- Introduce secure/admin activation for workflow changes.
- Treat secure enforcement as an experimental opt-in feature until the flow is proven in real use.
- Make non-opted-in repos warning-only rather than brittle or blocking by default.
- Keep the public model to two user-visible states: `warning-only` and `enforced`.
- Make workflow and transition diffs intelligible to humans.

Out of scope:
- Full ledger implementation for every event (covered by the signed-state epic).
- OS-backed keystore modernization before the core warning-only rollout is settled.

## Acceptance Criteria
- [ ] All child tickets under this epic are completed and Docket separates editable workflow proposals from approved runtime policy.
- [ ] Agents can continue normal work while secure and privileged controls remain opt-in, warning-only when not enabled, and consistently enforced when a team opts in.
- [ ] Workflow configuration remains customizable without letting agents weaken active policy mid-run once enforcement is explicitly enabled.
