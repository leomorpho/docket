# TKT-179: Add a keystore provider interface with an encrypted file backend

## Description
Create the first portable keystore implementation.

Context:
- Keep the design cognitively simple: one contract, one file-backed implementation, no biometrics yet.
- The key material should be suitable for later signature-based verification. An Ed25519-style signing key is an acceptable default unless implementation constraints strongly suggest another equally simple signing primitive.

What to do:
- Define a `KeystoreProvider`-style interface.
- Implement an encrypted file backend under `DOCKET_HOME`.
- Store at least: device/private signing key handle or bytes, trusted signer metadata, and repo anchor namespace entries.
- Make the storage versioned so future migrations are possible.

Likely areas:
- new package under `internal/security` or similar
- CLI wiring where secure/admin commands will acquire the provider

Out of scope:
- OS keychain integration.
- Remote/SaaS key custody.

## Acceptance Criteria
- [ ] A keystore contract exists and the initial implementation uses an encrypted file backend rooted at DOCKET_HOME.
- [ ] The backend stores private signing material outside the repo and exposes only the operations needed by later security tickets.
- [ ] Unit tests cover create load unlock read write and malformed-keystore handling without an OS-specific secret store.
