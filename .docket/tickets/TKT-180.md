# TKT-180: Add password-based secure mode with TTL and explicit privileged confirmations

## Description
Implement the first human approval surface for Docket.

Context:
- The user prefers password-based secure mode for now instead of Touch ID or Face ID.
- This should be implemented so a biometric or OS-keychain adapter can be added later behind the same contract.

What to do:
- Add a secure-session concept with unlock time, expiry, and lock/revoke behavior.
- Require secure mode for privileged operations only; agent-safe commands must continue to work without it.
- Prompt for confirmation before each privileged mutation rather than treating unlock as blanket approval.

Likely areas:
- new admin/security command layer
- future UI hooks for secure mode
- keystore package from the previous ticket

Out of scope:
- Biometrics.
- Multi-user auth flows.

## Acceptance Criteria
- [ ] Docket can enter and exit secure or admin mode by unlocking the local keystore with a password and the unlock expires after a short TTL.
- [ ] Every privileged action requires an explicit confirmation step that identifies the repo ticket and action.
- [ ] Tests cover unlock expiry relocking and rejection of privileged actions when secure mode is inactive.
