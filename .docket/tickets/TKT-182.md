# TKT-182: Add key enrollment, rotation, and recovery primitives for local-first identities

## Description
Prepare the local-first key model for future growth.

Context:
- The product target is still single-user, but the architecture should not dead-end when the user changes machines or later adds a SaaS backend.
- Keep this ticket focused on primitives and local metadata, not on a hosted identity service.

What to do:
- Define identity metadata and device-key metadata.
- Add commands or internal APIs to enroll a new device key and revoke/rotate an old one.
- Make sure the repo and `DOCKET_HOME` store enough information to re-establish trust after a machine migration.

Out of scope:
- Team roles and full multi-user collaboration.

## Acceptance Criteria
- [ ] A single-user installation can export public identity metadata enroll another device key and rotate or revoke an old key without rewriting repo history.
- [ ] The local trust model supports machine migration and keystore loss recovery through explicit flows documented by this ticket.
- [ ] Tests cover enrollment duplicate-key rejection rotation or revocation metadata updates and at least one recovery path.
