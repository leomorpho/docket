# TKT-183: Add a signed workflow.lock format for approved runtime policy

## Description
Create the runtime workflow artifact that the rest of the governance model will trust.

Context:
- The repo should remain config-driven, but active runtime policy must be pinned and reviewable.
- The lock file should be human-readable enough for Git review while still being treated as untrusted input until verified.

What to do:
- Define the file format and metadata fields.
- Include enough information to pin state transitions, prompt packs, and future model-routing decisions.
- Keep the format stable and versioned.

Likely areas:
- `internal/ticket/config.go`
- new workflow/policy packages
- CLI/admin commands for lock generation

Out of scope:
- Activation gating (covered by the next ticket).

## Acceptance Criteria
- [ ] Docket can materialize an approved workflow.lock that records the runtime workflow policy signer metadata and version hash.
- [ ] Editable workflow proposal files stay separate from the effective runtime lock so repo edits do not automatically change active behavior.
- [ ] Tests cover generation parsing signature metadata validation and rejection of malformed or stale lock files.
