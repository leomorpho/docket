# TKT-184: Gate workflow activation behind secure mode and pin runs to approved lock hashes

## Description
Turn workflow approval into a real gated step.

Context:
- Agents must be able to propose workflow changes, but they must not be able to self-activate weaker policy.
- Run pinning is essential so a ticket cannot start under one policy and quietly finish under another.

What to do:
- Add an activation flow that uses secure mode.
- Persist the approved workflow hash in the repo namespace under `DOCKET_HOME`.
- Ensure new runs record the active workflow hash.

Likely areas:
- workflow policy package from the previous ticket
- run/worktree metadata packages
- secure/admin command surface

## Acceptance Criteria
- [ ] A workflow proposal does not become active until a secure or admin action approves it and records the approved lock hash.
- [ ] Every new ticket run records the workflow lock hash it started under and live runs do not switch automatically after later proposal edits.
- [ ] Tests cover activation refusal without secure mode and run pinning across workflow edits.
