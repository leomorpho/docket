# TKT-185: Split docket into agent-safe and secure/admin capability surfaces

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-185` before committing. -->

## Description
Refactor the command and service layers so Docket has two authority levels.

Context:
- Today the UI uses the same generic CLI mutation path that an agent can also call, which collapses the human/agent trust boundary.
- The goal is one `docket` binary with two capability surfaces, not two unrelated CLIs.

What to do:
- Audit which operations are agent-safe versus privileged.
- Create a secure/admin entry path for privileged operations.
- Refactor shared business logic so both paths reuse the same lower-level services where appropriate.

Likely areas:
- `cmd/`
- web server integration points that shell out to the CLI
- workflow/domain services

## Acceptance Criteria
- [ ] Docket exposes a clear separation between agent-safe commands and privileged secure or admin operations without needing a second binary.
- [ ] Privileged operations are unreachable from the normal mutation path unless secure mode is active and the allowed command set is documented.
- [ ] Tests prove at least one previously flat mutation path now rejects privileged work through the agent-safe surface.
