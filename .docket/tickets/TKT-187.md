# TKT-187: Add reserved workflow semantics for review verification and terminal states

## Description
Keep the workflow configurable without letting configuration erase core safety rules.

Context:
- The user wants state names and transitions to be config-driven.
- Docket still needs a small set of non-negotiable semantics so security and review rules remain enforceable.

What to do:
- Extend workflow config to carry semantic annotations or equivalent metadata.
- Validate transitions against those semantics.
- Support a flow where agents can move work into review or verification states but not to closure.

Out of scope:
- UI diff views for these semantics.

## Acceptance Criteria
- [ ] Workflow config can customize state names and transitions while still declaring reserved semantics such as review human verification and terminal closure.
- [ ] Validation rejects workflow proposals that violate reserved invariants such as allowing non-human actors to close tickets or skipping required verification semantics.
- [ ] Tests cover a configurable state graph that maps to a ready-for-qa style verification phase without hardcoding the literal state name.
