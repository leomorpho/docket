# TKT-188: Extend ticket acceptance criteria for human verification preserved behavior and applicability

## Description
Upgrade acceptance criteria from a flat checklist into a workflow-aware contract.

Context:
- The current ticket schema only tracks simple checkbox AC items.
- The new design needs to encode whether a criterion is automated human-verified or a preserved-behavior promise and it needs room for verification steps.

What to do:
- Extend the ticket schema and rendering logic.
- Decide how the markdown stays readable while carrying richer structure.
- Use the new metadata to block or redirect transitions when human verification is required.

Likely areas:
- internal ticket types
- parser render and validation code under internal store local
- transition logic in workflow or state services

## Acceptance Criteria
- [ ] Tickets can represent structured criteria for automated checks human verification preserved existing behavior and applicability of changed areas.
- [ ] Validation and state-transition gating use the structured criteria so user-facing or CLI-facing changes can require explicit human verification steps before closure.
- [ ] Tests cover parsing rendering validation failures for missing human steps and at least one transition gate that refuses closure when required criteria remain unmet.
