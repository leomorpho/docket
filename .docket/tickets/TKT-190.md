# TKT-190: Build semantic diff views for workflow and state transitions

## Description
Make policy and transition changes reviewable by humans.

Context:
- Raw YAML or markdown diffs are not sufficient for understanding whether a workflow change weakened policy.
- The UI and CLI need a semantic summary layer for reviewers.

What to do:
- Define the semantic diff data model.
- Build the transformation from raw workflow or state changes into human-readable summaries.
- Surface the output in the most appropriate UI or CLI path first.

Dependencies to read first:
- workflow lock format
- structured workflow semantics
- ticket transition metadata

## Acceptance Criteria
- [ ] Docket can show a human-meaningful diff for workflow and transition changes instead of only raw text diffs.
- [ ] Ticket transition history shows actor from-state to-state reason relevant checks and linked workflow or runtime metadata in a reviewable format.
- [ ] Tests cover one semantic diff rendering case and one transition-history serialization case.
