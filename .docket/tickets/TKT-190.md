# TKT-190: Build semantic diff views for workflow and state transitions

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-190` before committing. -->

## Description
Make policy and transition changes reviewable by humans.

Context:
- Raw YAML or markdown diffs are not sufficient for understanding whether a workflow change weakened policy.
- The UI and CLI need a semantic summary layer for reviewers.

What to do:
- Define the semantic diff data model.
- Build the transformation from raw workflow or state changes into human-readable summaries.
- Surface the output in the most appropriate UI or CLI path first.

Dependencies to read first:
- workflow lock format
- structured workflow semantics
- ticket transition metadata

## Acceptance Criteria
- [x] Docket can show a human-meaningful diff for workflow and transition changes instead of only raw text diffs. : Implemented semantic workflow/state transition diff output: stale `docket workflow lock validate` now renders human-readable semantic change summaries (added/removed states and transition deltas) instead of only raw mismatch errors.
- [x] Ticket transition history shows actor from-state to-state reason relevant checks and linked workflow or runtime metadata in a reviewable format. : Added transition history surface in `docket show` (human/context/json) with actor, from_state, to_state, reason, checks, and linked runtime metadata (workflow lock hash, managed run branch/worktree when available).
- [x] Tests cover one semantic diff rendering case and one transition-history serialization case. : Added tests: cmd/TestWorkflowLockValidateRejectsStale (semantic diff rendering path), cmd/TestShowCmd_JSONIncludesTransitionHistory (transition-history serialization), internal/workflow/TestDiffWorkflowPolicyAndRenderHuman; all suites pass.
