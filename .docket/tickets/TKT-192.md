# TKT-192: Verify ledger chain integrity and signatures on privileged mutations

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-192` before committing. -->

## Description
Add verification at the point where privileged state matters.

Context:
- Repo files are always editable; the protection comes from verification rather than assuming files are untouchable.
- This ticket should focus on verification behavior rather than on defining the event format itself.

What to do:
- Hook ledger verification into privileged operations.
- Report failures clearly enough that a human knows whether the problem is corruption rollback or a missing approval.
- Keep verification scope tight enough that normal non-privileged reads remain fast.

## Acceptance Criteria
- [x] Before a privileged mutation is accepted Docket verifies that the relevant ledger chain and signatures are internally consistent. : Privileged operations enforce ledger verification through `SessionManager.RequireActive`, which calls `RepoNamespaceStore.VerifyAndAdvanceTrustedLedgerHead` before approvals are accepted.
- [x] Invalid or forged ledger entries are rejected with actionable errors rather than being silently ignored. : Invalid/forged ledger conditions are rejected with actionable errors (for example `trusted ledger guard failed`, rollback detection, malformed/signature validation failures) and are surfaced directly to callers.
- [x] Tests cover valid append broken chain bad signature metadata and duplicate or conflicting privileged events. : Coverage now includes valid append, rollback/broken chain, bad signature metadata, and duplicate/conflicting privileged approval events via `internal/security` tests including `TestTrustedLedgerHeadRejectsDuplicateOrConflictingPrivilegedApprovals`.
