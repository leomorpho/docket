# TKT-200: Add capability-tier model routing and a provider adapter contract

## Description
Build a stable abstraction for model selection.

Context:
- Providers ship new models frequently so Docket should not encode exact model names as its core abstraction.
- The workflow and policy layer should talk about capability tiers while adapters translate those tiers to concrete models.

What to do:
- Define the tier vocabulary.
- Define the provider adapter contract.
- Keep the implementation simple enough that new providers can be added without changing workflow semantics.

## Acceptance Criteria
- [x] Docket reasons about model selection in capability tiers such as cheap balanced strong and long_context instead of hardcoding provider model IDs in workflow logic. : Implemented in internal/workflow/model_routing.go with tier-first selection
- [x] A provider adapter contract exists so different agent runtimes can publish available models and relevant capabilities in a stable shape. : ProviderAdapter contract added with fixture-backed catalog mapping
- [x] Tests cover tier parsing validation and at least one adapter fixture that maps provider-specific models into Docket tiers. : Tests cover ParseCapabilityTier and provider mapping fixture
