# TKT-201: Map provider models to tiers and route work by task size risk and failure count

## Description
Turn capability tiers into practical model selection.

Context:
- The user is currently overpaying by using powerful models for trivial work.
- The routing logic should be mostly deterministic; use adapters and explicit heuristics instead of an LLM deciding what the provider catalog means on every run.

What to do:
- Define task features used for routing such as scope ambiguity user-facing risk and prior failure count.
- Map those features to tiers.
- Persist the selected tier and rationale in run metadata or an equivalent structure.

## Acceptance Criteria
- [ ] Given a provider adapter Docket can select a concrete model tier using deterministic routing rules based on ticket or task characteristics.
- [ ] Routing decisions are recorded in runtime metadata so humans can see which tier was chosen and why.
- [ ] Tests cover a cheap-task route a risky-task escalation and a fallback when the preferred tier is unavailable.
