# TKT-210: Define capabilities contract v1 for workflow, hooks, and skills

## Description
Context: Cross-platform behavior needs one canonical contract so adapter outputs and runtime behavior cannot drift.

Objective: Define a versioned capabilities contract and deterministic hash strategy used by CLI and MCP outputs.

Scope:
- Typed schema and model for workflow phases, hook events, and skill inventory.
- Contract versioning and hash generation semantics.
- Runtime write path at .docket/runtime/capabilities.json.
- Backward compatibility notes for future schema evolution.

Out of scope: Adapter-specific rendering details.

Delivery notes: Contract changes must be deterministic and easy to diff and review.

## Acceptance Criteria
- [x] Unit tests verify required fields, schema validation errors, and deterministic contract hashing for unchanged payloads. : go test ./internal/capabilities (TestValidateContractRequiresCoreFields, TestHashContractIsDeterministicForUnchangedPayload)
- [x] Integration test writes and reloads capabilities contract in a temp repo and confirms roundtrip parity. : go test -v ./internal/capabilities (TestWriteAndLoadRoundTrip writes and reloads .docket/runtime/capabilities.json in temp repo)
- [x] Validation evidence includes contract version and hash emitted in test output and summarized in a ticket comment. : go test -v ./internal/capabilities emitted version=1 hash=83b1d717f76b78f906932fc02bb10eabd4431f31fc421d656abbb304c32f2029
