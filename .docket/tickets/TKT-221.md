# TKT-221: Implement minimal lifecycle event set with stable payload schema

## Description
Context: A small stable event vocabulary is needed for observability and future extension without complexity.

Objective: Emit minimal lifecycle events with versioned payloads that adapters and tests can rely on.

Scope:
- Add run.start, phase.end, run.end, and tool.failure events.
- Define stable payload fields and versioning.
- Integrate event emission into relevant command flows.

Out of scope: Large event taxonomy or complex routing.

Delivery notes: Event ordering should be deterministic in integration harness runs.

## Acceptance Criteria
- [x] Unit tests validate payload schema and required fields for each emitted event type. : go test ./internal/lifecycle -run TestValidateEventSchemasAndRequiredFields && go test ./cmd -run TestLifecycleEventsSimulatedRunOrderAndSchemaFixtures
- [x] Integration test executes a simulated run and asserts expected event order and payload presence. : go test -v ./cmd -run TestLifecycleEventsSimulatedRunOrderAndSchemaFixtures (asserts deterministic order: run.start, phase.end, run.end, run.start, tool.failure, phase.end, run.end)
- [x] Validation evidence includes event log fixtures and schema-validation results referenced in comments. : Integration test writes fixture set under .docket/test-fixtures/lifecycle/{events.jsonl,schema-validation.json,event-order.txt} in fake repo and logs paths; schema validation uses lifecycle.ValidateEvent for each emitted event.
