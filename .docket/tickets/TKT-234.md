# TKT-234: Add machine-readable error envelope for mutating commands

## Description
Context: LLMs need field-specific remediation guidance, not generic command failures.

Objective: Standardize machine-readable error envelope across mutating commands.

Scope:
- Add error code, field, retryable, and suggested fix fields in json output.
- Normalize validation and transition errors to shared envelope shape.
- Keep human output concise while improving json diagnostics.

Out of scope: Full i18n error catalog.

## Acceptance Criteria
- [x] Unit tests verify stable error-envelope fields for validation transition and storage failures. : go test ./cmd -run "TestBuildMutationErrorEnvelopeValidationTransitionAndStorage" (stable envelope fields validated for validation, transition, and storage failures)
- [x] Integration test triggers representative failures and asserts structured json diagnostics are emitted. : go test -v ./cmd -run TestMutatingCommandsEmitStructuredJSONErrorEnvelope (triggers representative ticket apply validation failure and update transition failure; asserts structured json diagnostics)
- [x] Validation evidence includes failure fixtures with expected error_code and suggested_fix values. : Integration test writes failure fixtures with expected error_code and suggested_fix to .docket/test-fixtures/errors/{validation-failure.json,transition-failure.json}; fixture paths logged in verbose output.
