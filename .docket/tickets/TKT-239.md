# TKT-239: Epic: Screenshot Proof Evidence Pipeline for Ducca Tickets

## Description
Context: Ducca/LLM agents need a first-class way to attach screenshot evidence to tickets so proofs are durable, machine-readable, and visible in CLI and UI surfaces.

Objective: deliver an end-to-end screenshot proof pipeline: metadata schema, artifact storage, attach/list/remove commands, and UI rendering.

Decision: proof metadata uses one required narrative field `note` (why + what it proves) instead of separate reason/explanation fields.

Required proof fields (v1): `proof_title`, `note`, `added_at`, optional `captured_at`, image path/mime/size/checksum, and actor metadata.

Scope: local artifact handling, ticket linkage, API/CLI contract, output surfacing, and UI visibility.

Assumptions and non-goals (v1):
- local-first developer workflow using demo/mock data only (no production data handling requirements)
- reject unsafe/suspicious file inputs and path tricks as baseline safety
- defer advanced privacy controls and service-hosted upload auth until a later service mode
- defer max-proofs-per-ticket policy unless repo-bloat pressure appears in practice

Out of scope: cloud hosting, OCR, or external media indexing in v1.

Recommended execution order (critical path):
1) TKT-240 (proof metadata/storage base)
2) TKT-245 + TKT-246 (content-addressed blob policy + intake hardening)
3) TKT-241 (proof CLI add/list/remove)
4) TKT-242 + TKT-249 (CLI output surfacing + query/filter ergonomics)
5) TKT-243 (baseline UI rendering)
6) TKT-247 + TKT-248 (remove/gc semantics + lifecycle/ledger events)
7) TKT-250 (lazy-loading/scalable UI rendering)
8) TKT-244 (final end-to-end integration and quick-path validation)

Parallel governance stream: TKT-251 (privileged done-transition enforcement regression) under TKT-174.

Delivery notes: child tickets must be implemented TDD-first with deterministic fixtures and machine-readable evidence snapshots.
