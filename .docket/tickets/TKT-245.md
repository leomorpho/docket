# TKT-245: Implement content-addressed in-repo proof blob storage policy

## Description
Context: proof images should be durable and shareable via git history while minimizing binary duplication and reference breakage.

Objective: store proof image binaries in a content-addressed in-repo layout (hash-keyed) and reference them from ticket proof metadata.

Approach: write files under `.docket/proofs/by-hash/<sha256>.<ext>`, dedupe identical content, and store ticket-level references with stable proof IDs.

Assumptions: local-first tool; demo/mock data usage only; no external hosting in v1.

Likely paths: `internal/proof/*` (new), `internal/store/local/*`, proof metadata serializer, apply/validate helpers.

TDD plan: add failing unit tests for path derivation, dedupe behavior, and deterministic reference output before implementation.

Verify commands: `go test ./internal/... -run 'Proof|Blob'` and `go test ./...`.

## Acceptance Criteria
- [ ] Unit tests (written first) verify content-addressed path generation, dedupe on identical bytes, and deterministic blob references.
- [ ] Proof metadata references hash-addressed blobs without duplicate binary writes for identical images.
- [ ] Regression tests confirm existing proof operations remain stable across repeated add/remove/list cycles.
