# TKT-251: Investigate and fix `update --state done` behavior across warning-only and enforced modes

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-251` before committing. -->

## Description
Context: terminal-state gating appears inconsistent because `update --state done` is not consistently applying the expected warning-only or enforced behavior.

Objective: diagnose the policy path and restore deterministic behavior for terminal state transitions under the configured enforcement mode.

Approach: trace the update path, terminal-transition checks, secure session requirements, warning-only behavior, and configuration interactions; then enforce deterministic behavior with regression coverage.

Likely paths: `cmd/update.go`, privileged surface helpers, secure/session modules, transition tests.

TDD plan: write failing regression tests that reproduce missing prompt, missing warning, or incorrect enforcement first, then patch behavior.

Verify commands: `go test ./cmd -run 'Update|Privileged|Secure'` and `go test ./...`.

## Acceptance Criteria
- [ ] Regression tests written first reproduce and lock expected terminal-transition behavior for `update --state done` under warning-only and enforced modes.
- [ ] `update --state done` respects the configured enforcement mode consistently in interactive and automation flows, warning when warning-only and blocking only when enforced.
- [ ] Failure and warning output remain deterministic and actionable, with guidance on the active enforcement mode and any required secure session or flags.
