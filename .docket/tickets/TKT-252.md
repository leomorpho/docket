# TKT-252: Auto-transition review-ready tickets out of in-progress

## Description
Context:
Several tickets can end up stuck in `in-progress` even after implementation, AC evidence, and linked commits are present. This causes queue ambiguity and makes `docket start` choose poorly.

Objective:
Automatically move tickets that are review-ready from `in-progress` to `in-review` (or designated review semantic state), and prevent stale in-progress tickets from accumulating.

Why:
- Preserves truthful workflow state for humans and LLMs.
- Reduces repeated triage and cleanup passes.
- Keeps `docket start` and board views aligned with real execution status.

Scope:
- Add deterministic review-readiness evaluation in workflow and update paths.
- Apply transition automatically when readiness conditions are met.
- Add guardrails so tickets lacking required evidence remain non-review states with clear diagnostics.
- Respect configurable workflow semantics where review state names may be customized.

Likely paths:
- `cmd/update.go`
- `internal/workflow/service.go`
- `internal/ticket/state.go`
- `internal/hooks/core.go`
- `cmd/update_test.go`
- `cmd/start_test.go`

Out of scope:
- Human-only terminal transitions such as done and archived.
- Changing commit-linkage policy semantics.

TDD plan:
1. Add failing tests for tickets that meet readiness criteria but remain in-progress.
2. Implement readiness evaluation and auto-transition behavior.
3. Add regression tests for cases that must not auto-transition.
4. Validate with CLI integration tests for `start` `update` and automation mode surfaces.

Verify commands:
- `go test ./cmd -run TestUpdateAutoTransitionsReviewReadyTickets`
- `go test ./cmd -run TestUpdateDoesNotAutoTransitionWhenReadinessFails`
- `go test ./cmd -run TestStartAutoPreservesReviewTransitionBehavior`
- `go test ./...`

## Acceptance Criteria
- [ ] Unit tests are written first and fail initially for the stale in-progress case and then pass once automatic transition to review semantic state is implemented.
- [ ] Integration coverage proves deterministic automation behavior where review-ready tickets move to review and non-ready tickets remain in non-review states.
- [ ] Diagnostics clearly explain why a ticket did not transition including missing acceptance evidence missing commit linkage or policy gate failures.
