# TKT-281: Add hook event telemetry with CLI and MCP history surfaces

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-281` before committing. -->

## Description
Record recent hook advisory and blocking events with enough context to explain what happened, when it happened, and why, then expose that history through Docket itself. The feature is only complete if both humans and agents can inspect it through CLI and MCP surfaces, and the implementation should start from failing tests.

Likely paths:
- internal/lifecycle/
- cmd/status.go
- cmd/hook.go
- internal/mcp/handlers.go

Verify commands:
- go test ./cmd -run 'Test(StatusIncludesHookPolicyAndRecentBlockingEventsWhenDegraded|StatusIncludesQuietHookReadinessWhenHealthy|Hook)' -count=1
- go test ./internal/mcp -count=1
- go test ./internal/lifecycle -count=1

Out of scope:
- changing hook enforcement policy semantics
- adding a public hook execution surface beyond existing introspection and history views
- unrelated install/bootstrap hook wiring changes

## Acceptance Criteria
- [ ] Docket records recent hook advisory and blocking events with stable metadata and exposes concise history through both CLI and MCP surfaces.
- [ ] Unit tests and regression tests cover hook event capture, retention, CLI history output, MCP history output, and quiet behavior when no recent hook history exists.
