# TKT-296: Phase 2: Remove legacy artifact path fallback after migration window

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-296` before committing. -->

## Description
Context: During the .docket/local rollout we will support legacy artifact paths for compatibility, but that fallback must be explicitly removed after a defined migration window so path logic does not remain permanently split.

Objective: Add a controlled cutover that removes legacy artifact read/write fallback once migration preconditions are met, with clear operator messaging and deterministic behavior.

Approach:
1. Define cutover guard conditions (for example migration marker + doctor/check pass criteria).
2. Remove legacy read fallback paths from runtime/local artifact resolution behind the cutover condition.
3. Provide explicit failure/remediation messages when repositories are not yet cutover-ready.

Likely paths: internal/* artifact path helpers, cmd/upgrade.go, cmd/doctor.go, cmd/check.go, docs rollout notes.

Verify commands: go test ./... and focused migration/cutover regression tests.

Out of scope: introducing new artifact categories; changing unrelated runtime semantics outside artifact layout cleanup.

## Acceptance Criteria
- [ ] Go tests pass (run: go test ./...)
- [ ] Cutover removes legacy artifact fallback reads/writes behind explicit rollout preconditions and fails safely when preconditions are unmet.
- [ ] Regression tests prove no legacy-path dependency remains after cutover while preserving deterministic remediation guidance.
