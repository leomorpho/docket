# TKT-310: [Epic] Make workflow graph config-defined with fixed semantic roles

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-310` before committing. -->

## Description
Move Docket workflow behavior off hardcoded state-name assumptions and onto a versioned repo workflow config that defines the state graph, state names, transitions, and a small fixed set of semantic roles. Drive the work through small test-first tickets so the shipped default workflow preserves current behavior, custom repos can rename states or add intermediary states, and multiple states can share the same role without turning Docket into an unrestricted workflow DSL.

Scope:
- Config-defined workflow graph and state names.
- Fixed semantic roles such as intake, active, review, completed, and archived.
- Multiple states per role when the runtime behavior is the same.

Out of scope:
- Arbitrary user-defined runtime roles with no product semantics.
- Building a fully generic workflow engine for every imaginable lifecycle.

## Acceptance Criteria
- [ ] All child tickets under this epic land in dependency order with deterministic tests and a shared role-driven workflow evaluator or helper layer.
- [ ] The shipped default workflow reproduces current Docket behavior while allowing custom state names, configurable graph transitions, and multiple states per semantic role.
- [ ] Hardcoded workflow state-name assumptions are removed from core runtime paths that should follow config-defined graph and role semantics, while fixed product policy remains explicit in code.
