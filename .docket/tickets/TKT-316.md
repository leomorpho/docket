# TKT-316: Add parity and multi-state-per-role workflow fixtures for default and custom graphs

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-316` before committing. -->

## Description
Create regression fixtures that cover the shipped default workflow plus custom workflow shapes such as renamed active or review states, multiple intermediary states sharing the same semantic role, and non-blocking review-style states. Write fixture-driven tests first so later refactors can prove they honor the configured graph and fixed role semantics without silently regressing the default workflow.

Examples to cover:
- A renamed flow such as queued -> building -> qa -> shipped.
- A multi-active flow such as ready -> coding -> testing -> review where both coding and testing carry the active role.
- A review state that no longer blocks dependents.

## Acceptance Criteria
- [ ] Tests are added first to exercise default-workflow parity and at least two custom workflow shapes, including one with multiple states sharing the active role and one where a review state does not block dependents.
- [ ] Workflow fixtures are reusable across evaluator, CLI, MCP, and UI tests so downstream tickets can assert the same graph and role semantics everywhere.
- [ ] Go tests pass for the fixture and regression harness changes.
