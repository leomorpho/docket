# TKT-317: Refactor ticket transition validation to use the shared workflow evaluator

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-317` before committing. -->

## Description
Switch ticket transition validation off hardcoded state logic and onto the workflow evaluator so allowed transitions are determined entirely by config-defined semantics and state roles. Implement it from failing tests first and keep error messages explicit for invalid transitions under custom workflows.

## Acceptance Criteria
- [ ] Tests are added or updated first to validate transition enforcement through the workflow evaluator for both the default workflow and at least one custom workflow shape.
- [ ] Ticket transition validation uses config-backed evaluator decisions instead of hardcoded state-name rules.
- [ ] Go tests pass for transition validation changes.
