# TKT-338: Define wrapped-key keystore format and protector abstraction

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-338` before committing. -->

## Description
Optional phase 2 design work for a wrapped-key keystore model that separates keystore payload encryption from the protector used to unwrap the local data key. Do not implement this until the simpler warning-only versus enforced rollout is stable and clearly worth extending.

## Acceptance Criteria
- [ ] Failing tests are added first for protector abstraction behavior, wrapped-key keystore metadata, and unsupported-provider failure cases.
- [ ] Docket defines a stable protector abstraction and wrapped-key keystore format that no longer couples keystore encryption directly to a user password.
- [ ] Go tests pass for the abstraction and format changes.
