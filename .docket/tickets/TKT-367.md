# TKT-367: Refactor workflow service start and finish transitions to resolve active and review targets from config roles

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-367` before committing. -->

## Description
The workflow lifecycle manager still hardcodes `in-progress` for start and `in-review` for finish in `internal/workflow/service.go`. Write failing tests first, then move start and finish target selection onto workflow roles and config-defined transitions so custom state names work without breaking claim and worktree lifecycle behavior.

Likely paths:
- internal/workflow/service.go
- internal/workflow/commands.go
- internal/workflow/*_test.go
- internal/ticket/config.go

Verify commands:
- go test ./internal/workflow -run 'Test.*(Start|Finish|CustomWorkflow)'
- go test ./internal/workflow ./cmd

Out of scope:
- Changing privileged done policy.
- Reworking unrelated VCS or claim mechanics.

## Acceptance Criteria
- [ ] Tests are added first to prove StartTask and FinishTask work with at least one custom workflow shape that does not use the default state names.
- [ ] Workflow start and finish transitions resolve configured active and review targets from workflow semantics instead of literal state names.
- [ ] Go tests pass for the workflow service lifecycle paths after the refactor.
