# TKT-369: Refactor validation quality rules to derive handoff and activity expectations from workflow roles

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-369` before committing. -->

## Description
Validation still assumes handoff is required only for literal `in-review` and `done`, and that comment quality warnings apply only to literal `in-progress`. Write failing tests first, then derive those expectations from the configured graph plus fixed workflow roles so renamed workflows and intermediary active or review states preserve the same quality gates without depending on default state names.

Likely paths:
- internal/store/local/validate.go
- internal/store/local/validate_test.go
- internal/ticket/config.go

Verify commands:
- go test ./internal/store/local -run 'Test.*(Validate|CustomWorkflow|Handoff|Comments)'
- go test ./internal/store/local ./internal/ticket

Out of scope:
- Changing the warning thresholds themselves.
- Broad ticket quality policy redesign beyond role-driven state lookup.
