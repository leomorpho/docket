# TKT-370: Refactor check status and lock surfaces to derive active-work semantics from workflow roles

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-370` before committing. -->

## Description
Several runtime surfaces still equate active work with the literal `in-progress` state, including checker rules, status output, and lock heuristics. Write failing tests first, then move those surfaces onto shared graph-and-role helpers so custom active state names and multi-step active flows still behave as active work everywhere Docket reasons about current execution.

Likely paths:
- internal/check/rules.go
- cmd/status.go
- cmd/locks.go
- related *_test.go files
- internal/ticket/config.go

Verify commands:
- go test ./internal/check ./cmd -run 'Test.*(Status|Lock|CustomWorkflow|Checker)'
- go test ./cmd ./internal/check

Out of scope:
- Reworking lock storage format.
- New status UX unrelated to workflow role resolution.

## Acceptance Criteria
- [ ] Tests are added first to prove check status and lock behavior still works when the configured active-work flow uses non-default state names or multiple states sharing the active role.
- [ ] Active-work detection in checker status and lock surfaces uses shared graph and workflow-role helpers instead of literal `in-progress` checks.
- [ ] Go tests pass for the affected runtime surfaces.
