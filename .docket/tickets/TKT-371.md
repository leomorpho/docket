# TKT-371: Refactor wrap-up and auto-review guidance to use config-defined workflow roles and target commands

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-371` before committing. -->

## Description
Wrap-up readiness and auto-review guidance still assume the active state is `in-progress` and the review state is `in-review`, including human-facing next-step commands. Write failing tests first, then switch these surfaces to role-driven helpers so custom workflows produce correct readiness checks and guidance strings.

Likely paths:
- cmd/wrap_up.go
- cmd/auto_review.go
- cmd/wrap_up_test.go
- cmd/start_quickstart.go
- internal/ticket/config.go

Verify commands:
- go test ./cmd -run 'Test.*(WrapUp|AutoReview|CustomWorkflow|Quickstart)'
- go test ./cmd

Out of scope:
- New wrap-up policy checks unrelated to workflow naming.
- Changing the human-only done policy.

## Acceptance Criteria
- [ ] Tests are added first to cover wrap-up readiness and auto-review guidance for a custom workflow that uses non-default active and review state names.
- [ ] Wrap-up and auto-review use config-defined workflow roles to determine readiness and render next-step commands instead of hardcoded state names.
- [ ] Go tests pass for the updated guidance and readiness paths.
