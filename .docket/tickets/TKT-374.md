# TKT-374: Clean up workflow comments, tests, and examples that imply fixed state names

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-374` before committing. -->

## Description
Audit remaining code comments, helper descriptions, tests, and embedded examples that still say Docket always moves tickets to literal states like in-progress, in-review, or done when the implementation is now role-driven. Keep explicit default-workflow examples where useful, but relabel them as defaults and update misleading wording so future changes do not reintroduce state-name assumptions.

## Acceptance Criteria
- [ ] Misleading code comments and helper descriptions in runtime paths are updated to describe role-driven behavior instead of fixed state names.
- [ ] Tests and example fixtures that intentionally use default states are clearly scoped as default-workflow examples, while generic behavior tests use config-aware wording where appropriate.
- [ ] A targeted grep or equivalent check shows no remaining high-signal runtime comments/examples that incorrectly present default state names as product invariants.
