# TKT-376: Implement run-next as a serial loop that selects the next runnable ticket and repeats until exhausted

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-376` before committing. -->

## Description
Add the minimal serial orchestrator on top of the modular one-ticket run components. This command should repeatedly choose the next runnable ticket, invoke the one-ticket run flow, and stop cleanly when there is no more runnable work or when a blocking failure needs human attention.

This ticket should compose the selector and orchestrator layers. It should not absorb Codex launch details, raw output parsing, or final validation internals.

Likely paths:
- `cmd/`
- `internal/agentrun/`
- `internal/workflow/`

Verify commands:
- `go test ./cmd ./internal/...`
- `docket validate TKT-376`

Out of scope:
- Parallel execution.
- Reviewer-loop implementation.
- Process-control details that belong in the Codex adapter or run monitor.

Keep this intentionally serial and deterministic.

## Acceptance Criteria
- [ ] Tests cover next-ticket selection, dependency-aware stop conditions, and serial execution across multiple runnable tickets without parallel overlap.
- [ ] Docket exposes a small selector or orchestration layer that can repeatedly pick the next logical runnable ticket, invoke the one-ticket runner, and continue until no runnable work remains or a blocking stuck or failed outcome is hit.
- [ ] The loop leaves a clear machine-readable and human-readable summary of what ran what completed and why the cycle stopped.
