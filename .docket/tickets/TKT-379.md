# TKT-379: [Epic] Build serial Codex ticket execution with fresh sessions and a bounded review loop

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-379` before committing. -->

## Description
Create a Docket-native orchestration path that can pick one runnable ticket at a time, launch a fresh Codex session in a fresh worktree, drive the ticket to a deterministic machine-readable result, and then either advance to the next ticket or stop with a clear stuck reason.

Scope:
- Single-ticket serial execution only.
- Fresh session and fresh worktree for every implementation run.
- Docket-owned orchestration, selection, validation, and state transitions.
- Modular internal boundaries between run contract, agent adapter, monitor, validator, selector, and orchestrator layers.
- Simple structured outputs from Codex instead of parsing free-form conversation state.
- A capped optional reviewer/fix loop after the core implementer path works.

Likely paths:
- cmd/
- internal/workflow/
- internal/store/
- internal/git/
- internal/security/

Verify commands:
- go test ./cmd ./internal/...
- docket validate <child-ticket>

Out of scope:
- Parallel ticket execution.
- Long-lived multi-ticket agent memory.
- Planner/scout/debugger role explosion in v1.
- Generic support for every external agent framework before the Codex path is reliable.

## Acceptance Criteria
- [ ] All child tickets land in dependency order and produce a reliable one-ticket-at-a-time Codex execution path owned by Docket.
- [ ] The implementation lands with clean internal package boundaries so Codex process launching, output parsing, validation, ticket selection, and orchestration remain testable and replaceable modules rather than collapsing into cmd code.
- [ ] Each implementation run uses a fresh session and fresh worktree, ends in a machine-readable terminal result, and is revalidated by Docket before ticket state advances.
- [ ] The shipped workflow remains intentionally small: serial execution first, with at most one optional reviewer pass and one capped fix-review loop.
