# TKT-381: Add run monitoring, RESULT parsing, and timeout handling for Codex-managed ticket runs

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-381` before committing. -->

## Description
Build the runtime observer for the single-ticket Codex runner. Docket should tail process output, detect the structured RESULT line, handle timeouts or abnormal exit, and classify the run into a small deterministic set of terminal outcomes.

This ticket should own monitoring and terminal-result parsing only. It should consume output from the agent adapter and emit structured run outcomes without taking over launch, validation, or ticket-state authority.

Likely paths:
- internal/agentrun/
- internal/security/
- cmd/

Verify commands:
- go test ./cmd ./internal/...
- docket validate TKT-381

Out of scope:
- Semantic parsing of the whole transcript.
- Rich planner/reviewer status taxonomies beyond the small terminal state set.
- Final ticket validation and state transitions.

## Acceptance Criteria
- [ ] Tests cover successful RESULT parsing plus failure cases including missing result malformed result and no-output timeout handling.
- [ ] Docket exposes a run monitor that classifies the running Codex process deterministically to done stuck or failed based on structured output and process outcome.
- [ ] The implementation does not require parsing the full conversation as the source of truth; the structured RESULT contract is authoritative.
