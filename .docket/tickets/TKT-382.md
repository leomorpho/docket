# TKT-382: Define modular agent-run package boundaries and interfaces before orchestration implementation

<!-- docket: Direct edits are allowed. After editing this ticket, run `docket validate TKT-382` before committing. -->

## Description
Define the internal package boundaries for the new agent-run feature before the implementation tickets land. The goal is to keep the orchestration path testable and replaceable by separating run contract types, Codex process launching, output monitoring, validation, ticket selection, and top-level orchestration into small modules with clear authority boundaries.

Likely paths:
- internal/agentrun/
- internal/workflow/
- cmd/

Verify commands:
- go test ./cmd ./internal/...
- docket validate TKT-382

Out of scope:
- Full ticket execution behavior.
- Reviewer-loop implementation.
- Supporting multiple agent backends beyond the boundary definition needed for Codex.

## Acceptance Criteria
- [x] Tests or compile-time seams lock a small package map for agent-run orchestration so process launching, monitoring, validation, selection, and orchestration do not collapse into cmd code or a single package. : Added compile/runtime seam assertions in internal/agentrun/interfaces_test.go including explicit resumable-boundary checks for codex adapters.
- [x] Docket exposes clear interfaces between run contract, agent adapter, run monitor, final validator, and orchestrator layers, with Codex-specific logic isolated behind an adapter boundary. : Introduced dedicated codex.SessionRunner implementing ResumableAdapter while codex.Runner remains non-resumable adapter; codex-specific behavior stays behind adapter boundary.
- [x] Later implementation tickets can build against these seams without introducing circular dependencies or mixing ticket-state authority with process-control code. : Seam split removes ambiguity for later orchestration/health-check code paths and prevents mixing process-control capabilities into generic adapter flows.
